# Learn by Doing

**Created:** 2026-03-24 **Implementation Plan:**
docs/implementation-plans/2026-03-24-2005-learn-by-doing.md

## Overview

**What:** A `/learn-by-doing` skill — an alternative to `/build` that uses the same
implementation plan but flips the roles. The user writes all the code; Claude acts as a
tutor using evidence-based learning techniques forked from the
[learning-opportunities](https://github.com/DrCatHicks/learning-opportunities) skill
(CC-BY-4.0, Dr. Cat Hicks).

**Why:** The current workflow (design → plan → build) produces good output and teaches
through dialogue, but the user can end up out over their skis — especially when working
with unfamiliar technologies chosen specifically to gain experience. `/build` optimizes
for output velocity; `/learn-by-doing` optimizes for hands-on skill development while
still producing working code against the plan.

The design and plan phases are what make this work differently from generic tutoring. By
the time `/learn-by-doing` starts, the spec is fully scoped — like a lead engineer handing
off a spec to another engineer. Claude-as-tutor doesn't have to split attention between
figuring out _what_ to build and helping the user learn _how_ to build it. The what is
settled. All guidance focuses on the how.

**Type:** Feature

## Design Decisions

### Relationship to learning-opportunities

Fork learning-opportunities — carry forward the learning facilitation core (exercise
types, pause-for-input protocol, facilitation guidelines, PRINCIPLES.md), replace the
trigger/workflow layer with plan-driven task progression

### What carries forward from learning-opportunities

**Carried forward (largely verbatim):**

- Exercise types: Prediction → Observation → Reflection, Generation → Comparison, Trace
  the Path, Debug This, Teach It Back, Retrieval Check-in
- Core principle: Pause for input — end message after question, no hints, no suggested
  responses, no multiple questions
- Facilitation guidelines: ask before engaging, honor response time, adjust difficulty
  dynamically, embrace desirable difficulty, be direct about errors
- Techniques to weave in: elaborative interrogation, interleaving, varied practice
  contexts, concrete-to-abstract bridging, error analysis
- PRINCIPLES.md as a reference resource for judgment calls
- Hands-on code exploration: fading scaffolding ladder and "when to show code directly"
  guidelines. The primary context shifts — the user is already writing code, so these
  aren't the main interaction mode — but they apply when Claude points the user to
  existing patterns during dialogue or stuck escalation (e.g., "look at how
  `src/middleware/auth.ts` handles this same problem").

**Replaced:**

- "When to offer exercises" triggers — replaced by task-driven flow (every task gets a
  learning technique)
- Suppression conditions (declined once, max 2 per session) — don't apply; we're working
  through every task

### Stuck escalation

**Options considered:**

1. Immediate answer when asked — fastest but undermines learning
2. Fixed escalation ladder for every task — too rigid
3. Escalation as a safety net — normal flow uses learning techniques; escalation only
   activates when user is genuinely blocked after those techniques have been tried

**Decision:** Option 3. The escalation sequence (hint → pointed question → point to a
pattern in another file → show the answer) is a fallback, not the default interaction
mode. Normal task flow is Claude selecting and applying learning techniques, with dialogue
as the user codes.

### Learning log location

**Options considered:**

1. Inline in the implementation plan alongside the build log
2. Separate file in `docs/implementation-plans/`
3. Separate file in `docs/learning/`

**Decision:** Option 3. Keep the implementation plan clean as an execution record. The
learning log is a companion document at `docs/learning/YYYY-MM-DD-HHMM-feature-name.md`.
Future potential: learning logs can feed retrieval check-ins in the learning-opportunities
skill (out of scope for this feature).

### Learning log template

**Decision:** Start minimal. The template should capture at minimum: technique used per
task, what clicked, gaps surfaced, and a "revisit later" section. But the structure may be
better discovered through actual use than designed upfront. The first few sessions will
inform what's worth tracking.

### Task pacing

**Decision:** Same pause-between-tasks pattern as `/build`. In `/build` the pause is for
noting discoveries. In `/learn-by-doing` it serves double duty as a reflection point — the
user just worked through a task hands-on, and the pause is where learning context gets
captured before context-switching to the next task.

## Notes

### Future enhancements

- **Learning log → retrieval check-ins:** The "revisit later" section of learning logs
  could feed targeted retrieval check-ins in future `/learn-by-doing` or
  `/learning-opportunities` sessions. This is the most natural follow-on feature.
- **Learning log template refinement:** Let the template crystallize after a few real
  sessions rather than over-designing upfront.
- **Difficulty calibration:** Claude could adjust technique selection based on accumulated
  learning logs — if the user consistently nails predictions in a technology area, shift
  to harder techniques.
